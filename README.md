# UnscrambleGame

## Описание
UnscrambleGame — это мобильная игра-головоломка для Android, в которой игроку предлагается расшифровать перемешанное слово. Приложение отслеживает прогресс, подсчитывает очки за правильные ответы и автоматически завершает сессию после 10 вопросов, выводя итоговый результат с возможностью начать игру заново.

## 🛠 Технологии
- **Kotlin** — основной язык разработки
- **Jetpack Compose** — современный декларативный UI-фреймворк
- **ViewModel** — архитектурный компонент для управления состоянием UI и бизнес-логикой
- **StateFlow / MutableStateFlow** — реактивный поток для уведомления UI об изменениях состояния
- **JUnit 4** — фреймворк для написания и запуска Unit-тестов

## 📱 Функциональность приложения
- Отображение случайно выбранного перемешанного слова из встроенного словаря.
- Поле ввода ответа с проверкой правильности (регистронезависимое сравнение).
- Подсчёт очков: +20 за каждый правильный ответ.
- Индикация ошибки: красная подсветка и текстовое сообщение при неверном вводе.
- Кнопка «Пропустить» для перехода к следующему слову без начисления очков.
- Автоматическое сохранение состояния при изменении конфигурации (поворот экрана, смена темы).
- Диалог завершения игры с итоговой статистикой и кнопкой «Играть снова».
- Покрытие логики ViewModel Unit-тестами (happy path, error path, boundary cases).

# 1. Что такое ViewModel и зачем он нужен?
- Какую проблему решает: ViewModel отделяет бизнес-логику и состояние UI от визуальных компонентов (Activity/Composable). Он гарантирует, что данные не будут потеряны при пересоздании UI-контейнера, а также упрощает тестирование логики, вынося её в отдельные классы.
- Без ViewModel при повороте экрана: Android уничтожает Activity и создаёт её заново (configuration change). Все переменные, объявленные внутри Composable или Activity, сбрасываются к значениям по умолчанию, что приводит к потере прогресса игры.
## 2. В чём разница между MutableStateFlow и StateFlow? Зачем создавать два поля: _uiState и uiState? Что такое “backing property pattern”?
- Разница: MutableStateFlow позволяет изменять своё значение (emit/update), тогда как StateFlow — это интерфейс только для чтения.
- Два поля: _uiState (приватный, MutableStateFlow) используется внутри ViewModel для изменения состояния. uiState (публичный, StateFlow) отдаётся UI для подписки и чтения.
- Backing property pattern: Паттерн, при котором приватное изменяемое поле «скрывается» за публичным неизменяемым. Это предотвращает случайную модификацию состояния из UI-слоя, обеспечивает инкапсуляцию и поддерживает принцип однонаправленного потока данных (Unidirectional Data Flow).
## 3. Почему в ViewModel нельзя хранить Context или View?
- ViewModel живёт дольше, чем Activity или Composable. Если сохранить ссылку на Context или View, то после уничтожения экрана сборщик мусора (Garbage Collector) не сможет удалить старый экземпляр UI из памяти, так как на него всё ещё ссылается ViewModel.
- Последствие: Возникает утечка памяти (Memory Leak), которая со временем приводит к OutOfMemoryError и краху приложения.
## 4. Чем StateFlow отличается от remember и rememberSaveable? Когда лучше использовать каждый подход?
- remember: Сохраняет состояние только между перерисовками (recompositions). Теряется при повороте экрана. Подходит для кратковременного UI-состояния (например, анимация открытия меню).
- rememberSaveable: Сохраняет простые типы (Int, String, Parcelable) в Bundle. Переживает поворот экрана и смерть процесса. Подходит для простых полей ввода, когда ViewModel избыточен.
- StateFlow (в связке с ViewModel): Переживает изменения конфигурации, поддерживает сложные объекты, потокобезопасен и реактивен. Идеален для бизнес-логики и управления состоянием экранов. Не переживает смерть процесса без дополнительной настройки (SavedStateHandle).
## 5. Что такое data class в Kotlin и зачем он используется для UI State? Какие методы автоматически генерируются? Зачем нужна функция copy()?
- Data class — это специальный класс, оптимизированный для хранения данных.
- Автоматически генерируемые методы: equals(), hashCode(), toString(), copy(), а также геттеры для всех параметров конструктора.
- Зачем для UI State: Позволяет собрать всё состояние экрана в один иммутабельный (неизменяемый) объект. Это упрощает отладку, тестирование и предсказуемость изменений.
- Функция copy(): Создаёт новый экземпляр data class, заменяя только указанные поля, а остальные оставляя без изменений. Критически важна для работы со StateFlow, так как позволяет атомарно обновлять состояние: _uiState.update { it.copy(score = newScore) }.
## 6. Как работает collectAsState() в Compose? Почему UI автоматически обновляется при изменении StateFlow?
- collectAsState() — это Compose-функция, которая подписывается на поток StateFlow, преобразует его в объект State<T> и предоставляет текущее значение для чтения внутри Composable.
- Автоматическое обновление: Jetpack Compose отслеживает все чтения состояний (State). Когда StateFlow эмитит новое значение, collectAsState() уведомляет систему Compose о изменении, и она автоматически запускает recomposition (перерисовку) только тех компонентов, которые использовали это состояние.
## 7. Зачем разделять код на пакеты data и ui? Какие принципы архитектуры это поддерживает?
- Разделение ответственности (Separation of Concerns): Пакет data содержит модели, сущности и источники данных. Пакет ui (или presentation) содержит ViewModel, Composable-экраны и логику отображения.
- Поддерживаемые принципы: MVVM (Model-View-ViewModel), Clean Architecture, модульность. Это делает код более читаемым, упрощает командную разработку (одни работают с данными, другие с UI), позволяет писать изолированные Unit-тесты для логики без привязки к Android-фреймворку.
## 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)? Сохраняются ли данные в этом случае? Какие альтернативы есть для долговременного хранения?
- При process death (например, Android завершает процесс из-за нехватки оперативной памяти) приложение и все его объекты, включая ViewModel, полностью уничтожаются. Данные в ViewModel не сохраняются.
- Альтернативы для долговременного хранения:
- DataStore или SharedPreferences — для простых ключ-значение настроек.
- Room Database (SQLite) — для структурированных, реляционных данных.
- Файловая система или облачные хранилища (Firebase, Remote API) — для сложных медиа или синхронизации между устройствами.
- SavedStateHandle в ViewModel — позволяет сохранить небольшие примитивные данные, которые автоматически восстановятся после восстановления процесса системой.
